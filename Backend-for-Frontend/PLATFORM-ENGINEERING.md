# Platform Engineering — Backend-for-Frontend Pattern

## The BFF's Relationship to the Platform

The BFF sits at an unusual intersection in the platform model. It is owned and operated by a stream-aligned team — the frontend team that builds the client it serves. It is not a platform asset. Yet it depends critically on platform capabilities: auth middleware, observability instrumentation, request ID propagation, and contract test infrastructure. If those capabilities do not exist as self-service platform primitives, every BFF team rebuilds them independently, and divergence is guaranteed.

The platform team's job in a BFF architecture is not to own BFFs. It is to make spinning up a well-formed BFF fast enough that a frontend team can have a new BFF in production within one sprint, without writing any auth validation logic, logging schema design, or observability instrumentation from scratch.

This is the paved road model applied to BFF infrastructure: the platform team builds the road once, and every frontend team drives on it.

---

## The Paved Road for BFF Teams

| Without platform paved road | With platform paved road |
|---|---|
| Mobile team writes their own JWT validation logic | Auth middleware library installed with one import; JWT validation is automatic |
| Each BFF team designs their own logging schema | Structured logging schema is part of the scaffold; all BFF logs are uniform |
| Each BFF implements its own request ID generation | Request ID middleware is part of the scaffold; propagation is automatic |
| Each BFF writes its own circuit breaker implementation | Circuit breaker is a shared library primitive; BFF team configures thresholds, not implementation |
| Observability dashboard built per BFF by each team | Platform provides BFF dashboard template in Grafana; one variable change instantiates a dashboard for the new BFF |
| Contract test setup takes a sprint | Contract test scaffold is generated by the BFF scaffold tool; Pact broker is platform-managed |

The test of a good paved road: a frontend engineer who has never built a BFF before can have a production-ready BFF with auth, logging, contract tests, and observability in place in under three days. If it takes longer, the road has friction that the platform team should remove.

## The BFF Scaffold

The BFF scaffold is the primary platform deliverable for BFF enablement. It is a repository template (or CLI generator) that a frontend team runs once to create a new BFF project.

**What the scaffold provides:**

```
my-mobile-bff/
├── src/
│   ├── middleware/
│   │   ├── auth.js          # platform-owned; do not modify
│   │   ├── requestId.js     # platform-owned; do not modify
│   │   └── structuredLog.js # platform-owned; do not modify
│   ├── composition/
│   │   └── home.js          # BFF team owns: parallel domain calls, response projection
│   ├── routes/
│   │   └── index.js         # BFF team owns: endpoint definitions
│   └── cache/
│       └── index.js         # BFF team owns: TTL configuration
├── pact/
│   ├── consumer/            # BFF team owns: consumer-side contract definitions
│   └── pact.config.js       # platform-owned: Pact broker URL, auth
├── infra/
│   ├── task-definition.json # platform-owned template; BFF team sets resource sizing
│   └── alb-config.json      # platform-owned template
├── .github/
│   └── workflows/
│       ├── ci.yml            # platform-owned: lint, test, contract test, image build
│       └── deploy.yml        # platform-owned: deploy to ECS
├── Dockerfile                # platform-owned base image; BFF team does not modify
└── bff.config.yaml           # BFF team owns: client type, rate limits, upstream service URLs
```

**The key principle:** platform-owned files are not modified by the BFF team. They are consumed as dependencies or configuration templates. When the platform team updates the auth middleware library, all BFFs adopt the update by bumping a version in their package.json, not by accepting a PR to their auth.js file.

**What the BFF team owns:** composition logic, route definitions, cache TTL configuration, response projection (allowlists), and the consumer-side Pact contracts.

## Self-Service: Creating a New BFF for a New Client Surface

When a new client experience is launched (TV app, kiosk, partner portal), the frontend team follows this process to create the BFF:

**Step 1: Generate the scaffold**

```bash
# Platform-provided CLI
bff create --client-type tv --team tv-frontend --port 3003
```

This creates the project structure above, pre-configured with:
- The platform auth middleware imported at the correct version
- Structured logging schema loaded
- Pact broker configuration pointing to the platform's Pact broker
- CI/CD pipeline wired to the team's deployment credentials
- A placeholder `composition/home.js` with the correct import pattern and a TODO comment

**Step 2: Register the BFF with the platform**

The BFF team files a one-time registration that the platform team processes within 24 hours:

```yaml
# bff-registry.yaml (submitted as a PR to a platform-managed repo)
bff_name: tv-bff
client_type: tv
team: tv-frontend
service_account: tv-bff-sa
permitted_domain_services:
  - profile-service
  - catalog-service
  - recommendations-service
rate_limit_tier: standard
aud_claim: api.example.com/tv
```

The platform team creates the service account, configures the network policy to permit the BFF's address to reach the listed domain services, and provisions the Redis cache instance. After this one-time setup, the BFF team is self-sufficient.

**Step 3: Implement composition**

The BFF team writes the composition layer for their client's endpoints. They call domain services using the SDK the platform provides (which handles auth credential injection, circuit breaking, and request ID propagation automatically). They define the response projection for their client's rendering schema.

**Step 4: Write consumer-side contract tests**

For each domain service the BFF calls, the team writes a Pact consumer test that defines the minimum response shape the BFF requires. These tests run in CI. If a domain service changes its response schema in a way that breaks the BFF's contract, the Pact broker notifies both teams before the change reaches production.

**Step 5: Launch review**

Before the first production deploy, the security team reviews:
- Response allowlists for PII compliance
- Rate limit configuration relative to expected traffic
- JWT `aud` claim configuration

The platform team reviews:
- That platform-owned middleware files have not been modified
- That the service account's permitted domain services match what the composition layer actually calls

This review is time-boxed to 48 hours. It is not a recurring gate.

## Platform Contract

The platform team publishes and maintains a formal contract for BFF platform capabilities.

### What the platform provides

| Capability | SLA |
|---|---|
| Auth middleware library availability | Published to private npm registry; versioned; changelog maintained. Breaking changes require 30-day notice. |
| Request ID middleware | Part of scaffold; versioned with auth library |
| Structured logging schema | Platform-owned schema; BFF team adds fields but does not remove or rename platform fields |
| Pact broker | 99.9% availability; 30-day contract history retained |
| BFF scaffold tool | Updated within 30 days of any platform library change that requires scaffold updates |
| Observability dashboard template | Grafana dashboard template; instantiated per BFF by the team with one variable |
| BFF registry processing | New BFF registration processed within 24 hours on business days |

### What BFF teams are responsible for

| Responsibility | Owner |
|---|---|
| Composition correctness (right domain services, right fields) | BFF team |
| Response allowlist accuracy and completeness | BFF team |
| Cache TTL configuration | BFF team |
| Consumer-side Pact contracts | BFF team |
| BFF availability SLO | BFF team |
| Upstream domain service SLO budget (the BFF's calls are capped by domain service SLOs) | BFF team tracks; domain service team owns |
| Platform middleware library version updates | BFF team (within 90 days of new version release; immediate for security patches) |

## Signals the BFF Has Become a Platform Anti-Pattern

The BFF pattern relies on each BFF being independently owned by its frontend team. The following signals indicate the BFF has drifted from a stream-aligned asset toward a platform bottleneck:

| Signal | Root cause | Fix |
|---|---|---|
| Platform team is reviewing BFF PRs for content, not just library version compliance | Ownership confusion; platform team acting as gatekeeper for stream-aligned concerns | Platform team role is limited to: reviewing registration PRs, middleware library updates, and security launch reviews. Content PRs belong to the BFF team. |
| Shared BFF logic is accumulating in a "common-bff" library that isn't really per-client | Teams are solving composition duplication by creating a shared BFF, but the shared BFF now has client-specific conditionals | Extract to a shared composition library (pure orchestration, no response projection). If the shared library contains `if (clientType === 'mobile')`, it has become a single BFF serving multiple clients — anti-pattern. |
| BFF contains pricing or entitlement logic | A domain service returned pricing data and the BFF applied a discount rule rather than calling the Pricing service | Hard rule: any calculation that changes monetary value or access rights belongs in a domain service. BFF receives the final value and formats it. |
| Domain service teams are adding BFF-specific endpoints to their services | BFF teams are reaching into domain service internals rather than using domain APIs | Domain services should not know which BFF is calling. Domain APIs must be general. If a domain service has an endpoint named `/mobile-home-screen-data`, the BFF pattern has failed. |
| New BFF creation takes more than one sprint | Platform scaffold doesn't exist or is out of date; registration process is manual and slow | Fix the scaffold and the registration SLA. If it costs a sprint to start a BFF, teams will create one BFF and reuse it for multiple client types, which defeats the pattern. |
| BFF team carries on-call for domain service failures | BFF team has no fallback behavior; every upstream failure surfaces as a BFF failure | Add circuit breakers and partial response fallbacks. BFF on-call should be for BFF-layer issues (composition bugs, cache misses, auth failures), not for domain service outages. |

## Golden Path Integration Points

The BFF connects to the same platform capabilities as other services, but with BFF-specific integration patterns:

```
Identity Platform ──► Auth Middleware Library ──► BFF (JWT validation)
       │                        │
       ▼                        ▼
Pact Broker ─────────────► Contract Tests ──► Domain Services
       │                        │
       ▼                        ▼
Secret Management ──────► Service Account Credentials ──► Domain Service calls
       │                        │
       ▼                        ▼
Observability Platform ◄── Structured Logs + Traces ◄── BFF request handling
       │                        │
       ▼                        ▼
Cache Infrastructure ────► Redis (per BFF instance) ──► Response cache
```

Each of these integrations is automatic from the BFF team's perspective when the scaffold is used correctly. The BFF team does not configure the Pact broker URL — it is in the scaffold config. The BFF team does not provision Redis — it is provisioned as part of BFF registration. The BFF team does not write observability hooks — the structured logging middleware emits them automatically.

The platform team's ongoing responsibility is to keep these integrations working as underlying services change, and to update the scaffold when integration patterns evolve. The BFF team's responsibility is to keep their scaffold-derived configuration current.
