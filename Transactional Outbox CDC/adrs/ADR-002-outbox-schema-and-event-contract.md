# ADR-002: Standardize the outbox schema and event contract

## Status
Accepted

## Date
2025-10-16

## Context
The outbox table is the source of truth for all integration events published by the Orders service. Its schema determines the shape of every event that downstream consumers receive. A poorly designed schema creates three categories of problems:

**Insufficient identity information:** If events do not have a stable, unique identifier, consumers cannot implement idempotency. A consumer that processes the same event twice cannot tell it was a duplicate without a unique event ID.

**Missing routing information:** Without knowing the aggregate type and ID associated with an event, consumers that care about only some aggregates cannot filter efficiently.

**Payload versioning problems:** If the event payload changes its structure between service versions, consumers that have not updated to handle the new structure will fail. Without explicit versioning, there is no way for a consumer to know which payload format to expect.

The initial outbox table had none of these safeguards. When the first event schema change was deployed (adding a `discount_code` field to the `OrderCreated` event), one consumer crashed on the new field (it used strict deserialization that rejected unknown fields). Another consumer had no `event_id` to use for idempotency and had to implement its own deterministic ID derivation from the payload content -- which broke when the payload changed.

## Decision
The `outbox_events` table has the following schema:

```sql
CREATE TABLE outbox_events (
    event_id         UUID NOT NULL DEFAULT gen_random_uuid(),
    event_type       TEXT NOT NULL,           -- e.g., 'orders.order_created'
    event_version    INTEGER NOT NULL,        -- incremented on breaking payload changes
    aggregate_type   TEXT NOT NULL,           -- e.g., 'order'
    aggregate_id     TEXT NOT NULL,           -- the order ID
    occurred_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    payload          JSONB NOT NULL,
    trace_id         TEXT,                    -- distributed trace context
    correlation_id   TEXT,                    -- inbound request correlation ID
    status           TEXT NOT NULL DEFAULT 'pending',  -- pending | published
    published_at     TIMESTAMPTZ,
    PRIMARY KEY (event_id)
);

CREATE INDEX idx_outbox_status ON outbox_events (status, occurred_at)
    WHERE status = 'pending';
```

**Schema rules:**
- `event_id` is generated by PostgreSQL (`gen_random_uuid()`), not by application code. This ensures the ID is always present and unique regardless of which code path creates the outbox row.
- `event_version` starts at 1 and is incremented when the payload structure changes in a backward-incompatible way (field removed, field renamed, field type changed). Adding new optional fields does not increment the version.
- Payload JSON must be serializable without loss on round-trip; no floating-point approximations for currency amounts (use string-encoded decimals).

**Debezium routing:** Debezium is configured to route events to Kafka topics based on `event_type`: `orders.order_created` → topic `orders.order_created`, `orders.order_cancelled` → topic `orders.order_cancelled`. Topic names mirror `event_type` values, enabling consumers to subscribe to specific event types without filtering on the event body.

## Alternatives Considered

**Single payload column without structured schema:** Store all event data in a single JSONB `payload` column with no separate metadata columns. The payload contains all routing and identity information. Rejected because Debezium's topic routing logic needs structured columns for routing decisions; embedding routing information in the payload would require a custom Debezium transform.

**Separate outbox table per event type:** Each event type has its own outbox table (`order_created_outbox`, `order_cancelled_outbox`). Provides type-safe schema enforcement per event type. Rejected because it requires a schema migration for every new event type, adding friction to adding new integration events.

**Kafka-native schema with Schema Registry instead of outbox versioning:** Use Avro schemas registered in Confluent Schema Registry and enforce compatibility at the Kafka level. The outbox payload is always Avro-encoded. Rejected for the outbox layer because encoding the outbox row as Avro complicates direct database queries against the outbox table (DBA tooling cannot inspect Avro payloads natively). The outbox stores human-readable JSONB; schema enforcement happens at the consumer level via contract tests.

## Consequences

### Positive
- The `event_id` being database-generated ensures every outbox row has a unique, stable identifier regardless of the code path that inserted it; no application code needs to generate or track the ID
- The `status` column and `idx_outbox_status` index support a polling-based fallback relay (if Debezium is temporarily unavailable, a polling job can process `pending` rows) without schema changes
- Debezium's topic routing from `event_type` means consumers subscribe to well-named topics without needing to filter on event body contents

### Negative
- The `event_version` field requires discipline: developers who add a new required field to the payload without incrementing `event_version` create a silent schema incompatibility that consumers discover at runtime
- The JSONB payload stores event data denormalized; for events with complex payloads, the row size can be large (up to a few kilobytes for order events with many line items)

### Risks
- **Outbox table unbounded growth.** If published rows are not cleaned up, the `outbox_events` table grows indefinitely. Mitigation: a scheduled job deletes rows with `status = 'published'` and `published_at < NOW() - INTERVAL '30 days'` daily; see ADR-005 for retention policy details.

## Review Trigger
Revisit the JSONB payload approach if event payloads grow large enough to affect outbox table performance (row sizes exceeding 8KB consistently). At that point, moving large payloads to object storage with a reference URL in the outbox row may be warranted.
