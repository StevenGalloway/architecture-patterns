# AI Integration — Distributed Cache Invalidation

## Cache Invalidation as an AI Platform Primitive

Traditional cache invalidation operates on a simple premise: a key is stale when the data it represents has changed. For AI workloads, this premise breaks in three new ways:

1. **Model behavior changes without data changes.** A model upgrade makes all previous inference results stale, even if the input data is unchanged.
2. **Semantic equivalence is not exact-key equivalence.** Two different prompts that mean the same thing have different hash keys but should share a cache entry — and both should be invalidated together when the underlying knowledge changes.
3. **Index structures require a fundamentally different invalidation strategy.** A vector index cannot be partially invalidated; index staleness requires rebuild, not key deletion.

Each of these failure modes requires a distinct invalidation approach. The distributed cache invalidation infrastructure from the main pattern (NATS JetStream, L1/L2 eviction) handles cases 1 and 2. Case 3 requires a purpose-built strategy.

---

## 1. Model Update Invalidation

When a new model version is promoted to production, all cached inference results from the previous model version are stale. A response generated by model `sentiment-v1@1.2.3` is not a valid response from model `sentiment-v1@1.3.0`, even for the identical input — the model's behavior has changed.

**Key design for inference cache:**
```
inference:{model_id}:{model_version}:{request_hash}
```

Example: `inference:sentiment-v1:1.2.3:sha256:a3f7c9...`

The model version is embedded in the key, so inference results from different model versions occupy different key namespaces and cannot accidentally collide.

**Invalidation event on model promotion:**
```json
{
  "event_type": "model.promoted",
  "model_id": "sentiment-v1",
  "old_version": "1.2.3",
  "new_version": "1.3.0",
  "invalidate_all": true,
  "promoted_at": "2025-11-26T14:23:01Z"
}
```

**Consumer behavior:** on receiving `model.promoted` with `invalidate_all: true`, consumers must evict all L1 and L2 keys matching the pattern `inference:sentiment-v1:1.2.3:*`. Because model keys are namespaced by version, the `old_version` field gives the exact key prefix to sweep. The `new_version` entries are populated on first request after promotion — cold start is acceptable because the alternative (serving stale inference results from the wrong model) is not.

**Scale consideration:** if the inference cache is large (millions of entries from the previous model version), the Redis SCAN-and-DELETE operation after a model promotion creates a brief burst of Redis command volume. Batch the DEL commands using Redis pipelines in sets of 500–1,000 keys to avoid Redis command queue saturation.

## 2. Embedding Cache Invalidation

Embedding vectors are computed from document content. When the document changes, its embedding vector is stale — the vector no longer represents the current content, and nearest-neighbor searches using the stale embedding will return incorrect results.

**Key design for embedding cache:**
```
embedding:{model_id}:{document_id}:{chunk_index}
```

Example: `embedding:text-embedding-3-large:doc_456:chunk_0`

**Invalidation trigger:** a `document.updated` CDC event from the document store triggers invalidation of all embedding keys for that `document_id`, across all chunk indices and all embedding models that have cached it.

**Fan-out problem:** a single document update invalidates N chunks × M embedding models. For a document split into 20 chunks and cached by 3 embedding models, one document update generates 60 key eviction operations. This is manageable with the batched invalidation event design:

```json
{
  "event_type": "document.updated",
  "document_id": "doc_456",
  "tenant_id": "tenant_123",
  "invalidate_patterns": [
    "embedding:*:doc_456:*"
  ]
}
```

Consumers execute a Redis SCAN against `embedding:*:doc_456:*` and delete all matching keys in a single pass. The wildcard pattern handles all chunk indices and all model IDs simultaneously — teams do not need to know which models have cached a document to invalidate correctly.

**Why CDC is the right trigger:** write services that update document content may be decoupled from the embedding pipeline. Using CDC (reading the document store's write-ahead log) as the invalidation source ensures that every document update — regardless of which service performed the write — triggers embedding cache invalidation. This eliminates the "who publishes the event?" ownership gap described in TEAM-TOPOLOGY.md.

## 3. Vector Index Staleness

Approximate nearest neighbor indexes (HNSW, IVF) become stale as documents are added, removed, or updated. Unlike key-value cache invalidation — where evicting a key takes <1ms — vector index invalidation requires one of three strategies, each with a fundamentally different cost profile:

| Strategy | Cost | Freshness | Use case |
|---|---|---|---|
| **Full index rebuild** | High (minutes to hours for large indexes) | Complete freshness after rebuild | Low-frequency document changes, batch pipelines |
| **Incremental index update** | Medium (not supported by all vector DB implementations) | Near-real-time for HNSW with upsert support | Moderate-frequency updates, pgvector or Weaviate |
| **Soft invalidation** | Low (mark stale, rebuild asynchronously) | Stale results served with staleness signal until rebuild completes | High-frequency updates, latency-sensitive workloads |

The soft invalidation strategy deserves elaboration. When a document changes, rather than immediately rebuilding the index (which blocks search for minutes), the system:
1. Marks the affected document IDs as stale in a Redis set: `SADD vector_index:stale_docs:{index_id} doc_456`
2. Returns search results with a staleness header: `X-Index-Freshness: stale` for queries that touch stale documents
3. Enqueues an async index rebuild job
4. Clears the stale set after the rebuild completes

This is a different problem from traditional cache invalidation because there is no "evict the key" primitive for a vector index. The index is a derived structure that must be rebuilt from the source data — the invalidation event triggers a rebuild pipeline, not a cache eviction.

## 4. Semantic Cache Invalidation

Semantic similarity caches for LLM responses store answers to prompts and return them when a new prompt is semantically similar (cosine similarity > threshold). This works well for stable knowledge bases — but when the underlying knowledge base changes, exact-key invalidation is impossible because you cannot enumerate all prompts that were cached about a specific topic.

**The problem:** if a product's price changes from $49 to $79, semantic cache entries that answered questions like "how much does Product X cost?", "what's the price of Product X?", and "is Product X affordable?" all need to be invalidated — but these prompts have different hash keys, and the set of affected prompts is unbounded.

**Solution: tag-based invalidation at write time**

When caching an LLM response, tag the cache entry with the topics/domains it depends on:

```json
{
  "prompt_hash": "sha256:b8f3d...",
  "response": "Product X costs $49...",
  "model_id": "claude-3-5-sonnet",
  "model_version": "20241022",
  "cached_at": "2025-11-26T10:00:00Z",
  "topic_tags": ["product:product_x", "pricing", "tenant:tenant_123"]
}
```

Tags are stored in a reverse index: `semantic_cache:tag:product:product_x` → `[prompt_hash_1, prompt_hash_2, ...]`

**Invalidation on knowledge base change:**
```json
{
  "event_type": "product.price_updated",
  "product_id": "product_x",
  "invalidate_semantic_cache_tags": ["product:product_x", "pricing"]
}
```

Consumer behavior: look up all prompt hashes tagged with `product:product_x`, delete each from the semantic cache, and remove them from the reverse index. The tag-based approach makes it possible to invalidate an unbounded set of semantically related cache entries from a single, bounded event.

## 5. Multi-Model Cache Isolation

When multiple model versions are deployed simultaneously (e.g., during a canary release or A/B test), their inference caches must be strictly isolated. A response from model v1 must never be served by model v2's lookup, even if the prompt hash is identical.

**Key scheme:**
```
semantic_cache:{model_id}:{model_version}:{prompt_hash}
```

Example: `semantic_cache:claude-3-5-sonnet:20241022:sha256:c9d2e...`

Model version is embedded in the key namespace, so:
- Model v1 and v2 have entirely separate cache key spaces
- There is no possibility of a v2 lookup returning a v1 response
- When v1 is deprecated and v2 is fully promoted, the `v1` namespace is orphaned

**Garbage collection of orphaned model namespaces:**

Orphaned model namespaces (from deprecated model versions) are not invalidated via the standard eviction path — there is no `model.deprecated` event because there is nothing to "evict on write." Instead, the platform runs a scheduled cleanup job:

```
SCAN semantic_cache:claude-3-5-sonnet:20240229:* COUNT 100
  → DELETE matching keys in batches
```

This runs after the deprecation window closes (typically 7–14 days after the new version is fully promoted, to allow any in-flight requests against the old version to complete). The cleanup job is separate from the real-time invalidation path — it is a scheduled garbage collection, not an invalidation event.

**Canary deployment interaction:** during a canary release (e.g., 10% of traffic on v2, 90% on v1), both model namespaces are actively populated. Neither is stale — they are intentionally separate. The invalidation infrastructure does not need to coordinate across model versions during canary; each version maintains its own cache independently until traffic is fully shifted.
